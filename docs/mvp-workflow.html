MVP Paper Wiki Workflow

一条固定 workflow：把 raw paper 和你的阅读 insight 变成可演化的 Obsidian wiki， 再通过可靠 retrieval 反哺你的 research idea。

Product Boundary

这是 workflow，不是 agents 平台。LLM 参与每一步，但流程、gates 和落盘规则是预定义的。

Inputs

PDF、arXiv、DOI、本地 paper、Zotero 批注、你的读后 insight、一个新 research idea。

User Intent

ingest paper、补充我的理解、改进已有分析、找相关页面、基于 wiki 生成 idea。

Tools and Stores

paper parser、Zotero export、Obsidian CLI / Markdown files、search、graph/index/log updater。

Main Workflow

按 Anthropic 的 workflow 思路：routing、prompt chaining、parallel extraction、evaluator-optimizer、gated write。

predefined path
1

Route Request

判断请求类型：读新 paper、加入批注、修正分析、检索上下文，还是生成 idea。

2

Build Paper Model

并行抽取 problem、method、claims、experiments、limitations；把你的 insight 单独标成 user judgment。

3

Retrieve Context

用 index、frontmatter、lexical/BM25、graph links 找少量真正相关的 wiki pages。

4

Review Draft

生成可审查 draft；用 citation、schema、link、你的反馈检查是否能发布。

5

Publish and Feed Ideas

写入 Obsidian wiki，更新 index/log/graph；把可用结论反哺到 ideas 和 experiments。

Source Boundary

paper 原文事实、你的判断、wiki synthesis 必须分开。

Retrieval Quality

必须说明为什么选这些 pages，以及还缺什么证据。

Publish Gate

大改动先 draft；通过检查或用户确认后再写 canonical wiki。

Raw Store

`raw/papers`, `raw/zotero`, `raw/notes` 保持不可变。

Canonical Wiki

`wiki/papers`, `claims`, `methods`, `concepts`, `ideas`, `experiments`。

Control Files

`index.md`, `log.md`, `graph/*.jsonl`, templates, `AGENTS.md`。

Paper Analysis

一页可持续演化的 paper model：问题、方法、claims、实验、局限、你的批注、open questions。

Context Packet

面向当前问题的小上下文包：相关页面、选择原因、证据、冲突、缺口。

Idea Feedback

research angle、hypothesis、下一批 paper、实验草案、失败路径记忆。

MVP 成功标准：你给一篇 paper 和一些自己的批注，它能形成可追踪、可修改、可检索的 wiki 状态；之后你提出 idea，它能找回真正相关的 paper/claim/method，并给出有根据的下一步。
