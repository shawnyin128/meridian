# System Evaluation Optimization Loop

## Purpose

The optimization loop turns the System Evaluation Agent from a single-case reviewer into the evaluator inside Meridian's continuous optimization workflow.

It does not mutate canonical wiki pages by itself. It runs realistic use-wiki tasks, evaluates each context packet, aggregates repair buckets, and produces an optimization plan that points to generalized mechanisms.

## Round Inputs

- `eval cases`: JSONL scenarios with realistic research or coding queries.
- `retrieval context`: generated by `meridian wiki retrieve` through the loop runner.
- `selected pages`: top canonical pages loaded as read context for each case.
- `optional proposal/synthesis output`: future extension for write-back quality review.
- `audit summaries`: future extension for combining system checks with use-case evidence.

## Round Outputs

Each run writes:

```text
<out-dir>/
  <case-id>/
    case.json
    context.md
    context.json
    system-evaluation/
      system-evaluation.json
      system-evaluation.md
      judge-packet.md
  summary.json
  summary.md
  repair-buckets.json
  optimization_plan.md
  judge-packet.md

  comparison.md          # when --baseline-run is supplied
```

meridian wiki system-optimize-eval \

## Aggregation

The loop aggregates:

- pass / needs_refine / fail
- average, min, and max weighted score
- hard failures
- recurring findings
- repair bucket frequency
- lowest scoring dimensions
- top leverage candidates

Top leverage candidates are chosen from repair bucket frequency, hard-failure impact, needs-refine impact, and affected workflows.

## Repair Bucket To Optimization Path

- `retrieval_context`: query decomposition, corpus selection, or context packing.
- `retrieval_ranking`: result-type routing, section-aware scoring, or hard-negative suppression.
- `knowledge_layer`: method/topic/concept/claim/evidence consolidation or promotion.
- `provenance_schema`: source_papers, sources, evidence maps, and trace fields.
- `artifact_boundary`: product output contract and draft/version suppression.
- `source_quality_routing`: source-quality cleanup route versus scientific evidence route.
- `synthesis_quality`: write-back schema, synthesis refinement, or publish contract.
- `concept_layer`: prerequisite concept coverage and coding/debug sections.
- `claim_evidence_traceability`: claim/evidence promotion and provenance surfacing.
- `entry_contract`: Prompt/MCP compact output and canonical artifact boundaries.
- `personalization_boundary`: user insight labeling and source-fact separation.

## Convergence Use

The loop should be run before and after generalized fixes. A useful optimization round should show one or more of:

- fewer hard failures
- higher average or minimum score
- reduced recurring repair bucket count
- improved weak dimensions
- fewer case-level `needs_refine` decisions

When scores flatten and remaining findings are non-blocking residuals, the continuous optimization loop can stop and move to broader case coverage or sampled semantic judge review.
