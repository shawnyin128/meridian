Meridian Full Architecture

两个产品，一个研究系统：Paper Wiki 是 workflow 产品；Research Dev 是 agentic 开发插件。 Paper Wiki 可以作为 MCP delivery，Dev 通过稳定工具接口使用 wiki，而不是耦合 vault 内部结构。

Boundary

Wiki 维护 canonical research memory；MCP 暴露 context/search/read/write-back；Dev Agent 自主决策工具调用、诊断路径和 wiki lookup。

Wiki-only

只读 paper、Zotero 批注、维护 claim/method/idea，不进入代码仓库。

Dev-agent-only

只在代码仓库里做实验设计、sanity check、结果解释、复现诊断和实验记忆。

Integrated

Dev Agent 先向 Wiki 要 context packet，实验后用 write-back packet 更新研究记忆。

Implementation Order

先构建 Paper Wiki。Dev Agent 之后再接入，因为它的 wiki-gateway 能力依赖一个真实可用的 wiki backend。

1. Brainstorm

界定 prototype 边界、非目标、人工审查点和评测维度。

2. Prototype

做最小 Paper Wiki：ingest、user insight、context packet、draft/publish。

3. Human Review

人工检查是否真的符合 paper reading 和 idea refinement 需求。

4. Test Cases

大量案例写清问题和预期结果，但允许多条有效路径。

5. Evaluation

评估 artifact quality、provenance、retrieval、write boundary、failure honesty。

6. Refine

失败归因到 workflow、schema、retrieval、MCP/tool interface，再迭代。

Source-Grounded Principles

来自 LLM Wiki 指南、Anthropic agent engineering posts 和 followup 经验：原则进入系统边界，具体社区实现不照搬。

Simple First

固定任务用 workflow；不确定科研开发才用 agentic loop。

Progressive Disclosure

先 metadata / index / packet，再按需展开 page、source、tool schema。

High-Signal Context

返回选择原因、provenance、conflicts、gaps，而不是 retrieval dump。

Agent-Ergonomic Tools

MCP tools 要少、清晰、可评测，支持 concise/detailed 输出。

Reviewable Writes

先 propose/apply，再改 canonical wiki；判断点显式暴露。

Persistent Learning

把 query、失败实验、设计决策、stale claims 都沉淀回 wiki。

Impact Rollback

科研代码用 staged commits 保留可比较、可回退、可归因的时间轴。

Paper Wiki MCP Delivery

Paper Wiki 的本体仍是 raw sources + Markdown wiki + schema。MCP 是 delivery boundary，让 Dev Agent 和其他客户端通过稳定工具契约使用研究记忆。

Stable Boundary

客户端不需要知道 paper/claim/method 页面实际放在哪里，只请求 context、page、search、write-back。

Search

查 papers、claims、methods、ideas、experiments、results，并返回 provenance。

Context Packet

为科研问题或 coding task 选择小而准的 pages、证据、冲突和缺口。

Read Page

读取具体 wiki page、source summary、frontmatter 和引用关系。

Propose Write-back

接收实验/代码证据，生成可 review 的 wiki 更新建议，而不是静默改 canonical memory。

Research Dev Boundary

Dev 产品最低成立条件有三个：写出科研级代码；作为 wiki 的开发入口；善用 git checkpoint 支撑科研试错。

1. Research-grade Code

清晰、易读、易维护、易扩展；允许为 probe、ablation、variant isolation 保留有目的的冗余。

2. Wiki Gateway

开发中主动查 paper/wiki memory，把 prior work、method detail、failed path、user insight 用到代码和解释里。

3. Git Checkpoints

在 hypothesis、probe、ablation、result、risky refactor 边界建议阶段性 commit，支持 impact rollback。

Target Artifact

代码要显式保留 configs、seeds、metrics、output paths、probe hooks 和 variant boundaries。

Agent Freedom

允许 agent 自主读代码、跑命令、加诊断、查日志、决定下一步 sanity check 或 ablation。

Wiki Awareness

遇到 paper definition、method detail、prior failed path、metric claim、user insight 时主动请求 context packet。

Thin Guardrails

不套重流程；只要求 learning target、evidence plan、checkpoint prompt、write-back packet。

Paper Wiki Workflow

预定义 workflow。目标是稳定维护研究状态，并通过 MCP deliver 给不同客户端。

workflow product

Ingest Paper

raw paper、Zotero annotations、user insight 进入 draft analysis。

Evolve Analysis

基于反馈更新 paper model、claims、methods、open questions。

Retrieve Context

返回小而准的 context packet，而不是一堆搜索结果。

Maintain Memory

更新 index、log、graph、paper/claim/method/idea/result pages。

Context Packet

Paper Wiki MCP -> dev: 相关 pages、选择原因、证据、冲突、缺口、允许写入策略。

knowledge grounding

Write-back Packet

dev -> Paper Wiki MCP: 实验目的、命令/config、结果、解释、claim 影响、失败路线、下一步。

Research Dev Agent

Agentic plugin。目标是把自由探索转成 researcher 看得懂、可维护、可扩展、可解释的代码和证据。

agentic product

Classify Event

实验设计、sanity check、结果解释、实验记忆、复现诊断、普通 coding。

Inspect And Decide

读代码、查配置、查 wiki、选命令、决定 probe、ablation、diagnostic action。

Run Or Diagnose

调用 shell/tests/scripts，加 instrumentation，处理失败、恢复 partial results、记录证据。

Interpret And Next

判断结果是否支持 hypothesis，是否需要 control、ablation 或停止方向。

Shared Research Memory

两件产品共享同一套研究状态，但拥有不同控制模型。Wiki 负责 canonical memory；Dev Agent 负责把代码和实验过程变成可回写的证据。

Papers

论文理解、批注、隐藏实现细节。

Claims

支持、反驳、置信度、证据来源。

Methods

公式、实现、probe、ablation 关系。

Experiments

config、command、metric、run evidence。

Ideas

hypothesis、failed paths、next decisions。

Paper-grounded Ideas

想法不是凭空生成，而是从 paper、claim、method、failed path 中被 refine。

Experiment-ready Code Work

代码允许为 ablation/probe 保留冗余，但必须服务明确观察目标。

Persistent Research Learning

每次结果都回到 wiki：说明什么、不说明什么、下一步是什么。
